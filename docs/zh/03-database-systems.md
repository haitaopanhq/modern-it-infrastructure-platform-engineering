# 第 3 章：数据库系统

![数据库系统图解](../zh/diagrams/03-database-systems.png)

## 本章概述

本章的现实问题是：为什么数据库从一个“存数据的系统”，变成了业务增长、组织信任和 AI 能力的共同底座。

交易系统关心一致性，增长系统关心吞吐，分析系统关心时效，AI 系统关心检索和上下文。数据库的演进，表面上是从关系型到分布式、从 OLTP 到 OLAP、从结构化到向量化，本质上是企业在“正确、快速、便宜、可扩展”之间不断重新选择。

很多人以为，数据库就是 MySQL。会建表、会写 SQL、懂主从复制，好像就已经进入了“数据库世界”。这个判断在 Web 业务早期并不离谱：一个 MySQL 加一点缓存，确实可以支撑很多公司从零到一。

但 AI 时代之后，数据库体系已经彻底裂变。过去数据库最核心的问题是“数据能不能存下来”。现在问题开始变成：数据能不能实时流动，能不能支撑 AI 检索，GPU 能不能高速读取，OLTP 和 OLAP 能不能分离，向量搜索能不能毫秒级返回。

数据库因此不再只是业务系统背后的配角，而是 AI 基础设施里的核心层。它一边连接交易、一致性和权限，一边连接分析、语义检索和模型上下文。谁能组织数据，谁就能组织业务；谁能让数据进入模型上下文，谁就能把 AI 能力真正接进企业系统。

## 3.0 从单一数据库到数据基础设施层

过去很多公司是“一个 MySQL 打天下”：交易、日志、分析、搜索、报表，全塞进同一个系统。起步时这样最省事，团队少、数据少、功能少，单一数据库反而是最快路径。

问题出现在增长之后。慢查询开始影响交易，报表扫描拖垮线上库，日志写入冲击主业务，搜索需求绕着索引打补丁，主从复制变成所有扩容方案的默认答案。数据库并不是突然不行了，而是被迫承担了太多不同性质的工作负载。

现代架构开始把数据系统拆成多层：

| 数据层 | 主要任务 | 典型系统 |
| --- | --- | --- |
| 事务层 | 订单、账户、支付、库存等强一致业务 | MySQL, PostgreSQL |
| 缓存层 | 热点数据、会话、计数、限流 | Redis |
| 分析层 | 日志分析、报表、用户行为、宽表查询 | ClickHouse, DuckDB, Databend |
| 搜索层 | 关键词检索、全文索引、日志搜索 | Elasticsearch, OpenSearch |
| 向量层 | Embedding 检索、RAG、语义召回 | Milvus, Qdrant, pgvector |
| 数据湖仓层 | 历史数据、离线计算、跨系统治理 | Iceberg, Hudi, Delta Lake |

这就是数据库世界的 Kubernetes 化：不是所有能力都塞进一个引擎，而是把不同工作负载拆成可组合的数据基础设施层。事务归事务，分析归分析，缓存归缓存，向量归向量，最后由平台、网关、数据管道和治理体系把它们重新接成一个整体。

## 3.1 数据库分类

数据库分类不是为了记产品名，而是为了看清业务压力如何被写进系统形态。每一种数据库，都是某类业务矛盾的工程化结果。

### OLTP vs OLAP vs HTAP

| 类型 | 场景 | 特征 | 典型产品 |
|------|------|------|----------|
| OLTP | 事务处理 | 高并发、低延迟 | MySQL, PostgreSQL |
| OLAP | 分析处理 | 大数据量、复杂查询 | ClickHouse, Snowflake |
| HTAP | 混合负载 | 事务+分析一体化 | TiDB, SingleStore |

## 3.2 关系型数据库

### MySQL vs PostgreSQL

| 特性 | MySQL | PostgreSQL |
|------|-------|------------|
| 事务 | 支持 | 支持 |
| 复杂查询 | 一般 | 强大 |
| 扩展性 | 主从复制 | 分片、FDW |
| JSON 支持 | JSON 类型 | JSONB（原生） |
| 生态系统 | 丰富 | 丰富 |
| 适用场景 | Web 应用 | 企业级应用 |

MySQL 不会消失，但它不再万能。它仍然是大量业务系统最稳妥的事务底座，尤其适合成熟 Web 应用、订单系统、账户系统和中小规模业务数据。但当一个系统同时要求实时分析、全文搜索、向量召回、跨区域扩展和 AI 上下文组织时，MySQL 就不应该继续独自承担所有职责。

PostgreSQL 正在变得越来越像“数据库 Linux”。它的意义不只是更强的 SQL 能力，而是强大的扩展机制：PostGIS 让它进入地理空间系统，TimescaleDB 让它进入时序场景，FDW 让它连接外部数据源，pgvector 又让它进入 AI 检索链路。它不再只是一个关系型数据库，更像一个数据平台内核。

这也是为什么很多 AI 项目会从 PostgreSQL + pgvector + RAG 起步。小团队不一定一开始就需要独立向量数据库、复杂数据湖和完整检索平台，但需要一个能把业务数据、权限、SQL 查询和语义向量放在同一套治理边界里的基础底座。

### 核心概念

- **ACID**：原子性、一致性、隔离性、持久性
- **MVCC**：多版本并发控制
- **WAL**：预写日志
- **索引**：B+Tree、Hash、全文索引

## 3.4 向量数据库

### AI 时代的数据库

向量数据库专门用于存储和检索 Embedding 向量，支撑 RAG（检索增强生成）应用。它的出现说明数据系统正在从“记录事实”走向“组织语义”：企业不只是要保存数据，还要让模型能够找到、理解和利用数据。

过去搜索主要靠关键词。用户输入一个词，系统匹配倒排索引，返回包含这个词的结果。AI 时代之后，搜索开始从“词相同”转向“意义相近”。Embedding 把文本、图片、代码、日志和知识片段映射成向量，系统就可以根据语义距离找到相关内容。

这就是 Milvus、Qdrant、Weaviate、Pinecone 和 pgvector 爆发的原因。RAG 的核心不只是“把文档塞给大模型”，而是让模型在回答之前先从企业知识库、业务系统、工单、代码仓库和历史记录里找回上下文。未来很多系统的入口，可能不再是“你搜索什么”，而是“AI 判断你真正需要什么”。

向量数据库也带来了新的基础设施问题：Embedding 如何生成，索引如何更新，权限如何过滤，召回结果如何排序，过期知识如何淘汰，模型上下文如何防止泄露。语义检索看起来是 AI 功能，落到工程里仍然是数据系统、权限系统和治理系统的组合。

### 主流向量数据库

| 数据库 | 特点 | 索引算法 |
|--------|------|----------|
| Milvus | 开源，功能丰富 | HNSW, IVF |
| Pinecone | SaaS，性能高 | 专有 |
| Qdrant | Rust 实现，高性能 | HNSW |
| Weaviate | 图结构+向量 | HNSW |

### 向量检索流程

```
1. 数据导入
   文档 → 分词 → Embedding 模型 → 向量 → 存入向量数据库

2. 查询流程
   用户查询 → Embedding 模型 → 向量 → 向量数据库检索 → 返回结果
```

## 3.6 数据库选型决策树

```
开始
  │
  ├─ 是否需要事务？
  │    ├─ 是 → 关系型数据库
  │    │    ├─ 数据量 < 1TB → MySQL/PostgreSQL
  │    │    └─ 数据量 > 1TB → TiDB/CockroachDB
  │    │
  │    └─ 否 → 非关系型数据库
  │         ├─ 需要复杂查询 → Elasticsearch
  │         ├─ 需要高速缓存 → Redis
  │         ├─ 需要向量检索 → Milvus/Pinecone
  │         └─ 需要图关系 → Neo4j
  │
  └─ 是否需要分析能力？
       ├─ 是 → OLAP 数据库
       │    ├─ 实时分析 → ClickHouse/Druid
       │    └─ 数据湖 → Iceberg/Hudi
       │
       └─ 否 → 回到事务决策
```

这个决策树只能作为起点，不能当成最终架构答案。真实企业系统往往不是“选一个数据库”，而是设计一条数据路径。

一条典型的现代数据路径会这样展开：

```text
业务交易
  -> MySQL / PostgreSQL 保存事实
  -> Kafka / CDC 推动数据流动
  -> ClickHouse / Databend 支撑分析
  -> Redis 承接热点访问
  -> Milvus / Qdrant / pgvector 承接语义检索
  -> RAG / Agent / AI Gateway 消费上下文
```

在这条链路里，数据库已经从一个独立产品，变成数据面的一部分。交易库负责正确性，消息系统负责流动性，OLAP 负责分析速度，缓存负责访问效率，向量库负责语义召回，AI Gateway 和 Agent Runtime 负责把数据变成可执行的智能能力。

## 3.7 OLAP 重新崛起

OLAP 正在重新变得重要。ClickHouse、DuckDB、Databend 这几年快速流行，不是因为报表突然时髦，而是因为 AI 和数据分析时代之后，真正昂贵的东西变成了查询速度、数据新鲜度和计算成本。

ClickHouse 几乎已经变成新一代日志分析基础设施。它适合高吞吐写入、列式压缩和大规模聚合查询，正好对应可观测性、用户行为分析、风控分析和实时看板这些场景。DuckDB 则把本地分析能力带到开发者机器和数据文件旁边，让“轻量分析”不再必须先搭一整套大数据平台。Databend 代表的是云原生数仓方向，把对象存储、弹性计算和 SQL 分析重新组合。

OLAP 回归说明一个变化：企业不只想保存数据，还想尽快解释数据。过去分析是离线系统的事情，今天分析越来越接近业务现场。异常检测、实时指标、推荐系统、模型评估和 Agent 决策，都需要更快的数据反馈。

## 3.8 数据库进入 AI Native 时代

数据库已经不只是 CRUD。它正在变成 Data + Vector + AI + Agent 的组合层。

SQL 正在和 AI 融合。自然语言生成 SQL、SQL 查询解释、数据血缘分析、异常自动归因，都会让数据库从被动执行命令，变成能参与分析和决策的系统。

向量检索正在和 OLAP 融合。企业既要按语义找相似内容，也要按时间、标签、组织、权限、成本和业务指标过滤结果。只做向量相似度不够，只做结构化查询也不够，未来的数据系统会同时处理结构化事实、非结构化内容和语义向量。

实时流处理正在和 Agent 融合。Agent 不只读取数据库，还会触发任务、调用工具、更新状态、写回结果。数据库因此必须进入更严格的权限、审计、回滚和隔离体系。因为一旦 AI 能写数据，数据系统就不只是查询底座，而是自治软件的状态层。

这意味着数据库行业正在经历过去二十年最大的重构。很多人还停留在“会写 SQL 就够了”的时代，但现代数据库工程已经变成系统工程：事务、缓存、消息、分析、向量、权限、治理、成本和 AI 上下文必须一起设计。

## 本章收束

数据库演进从来不只是引擎升级。它背后是业务对一致性、性能、分析、检索、成本和智能化的持续拉扯。现代数据系统越来越像分布式基础设施，而不是单个数据库产品。

MySQL 和 PostgreSQL 仍然重要，但它们不再独自代表数据库世界。ClickHouse、DuckDB、Databend 代表分析速度，Redis 代表热点访问，Kafka 和 CDC 代表数据流动，Milvus、Qdrant 和 pgvector 代表语义检索，数据湖仓代表长期治理。AI Native 时代的数据库，不是某一个产品的胜利，而是数据基础设施层的重组。
