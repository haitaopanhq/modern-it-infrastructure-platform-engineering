## 本章概述

本章从《一图看懂账户与身份安全演进史》的主线展开：现代安全最大的变化，不是防火墙更厚，而是安全边界从“机器可信”“内网可信”迁移到“身份可信”“行为可信”。

真正完整的现代身份安全，是一套从用户身份、工作负载身份、云资源身份、CI/CD 身份，到密钥、策略、审计、供应链的长链路系统。它回答的不是“有没有登录”，而是：谁访问、用什么身份、拿什么凭证、是否可验证、是否可撤销、是否可审计。

过去一周，我们一直在讨论虚拟化、资源池、云平台与云原生架构。它们共同解决的是资源效率问题：如何让计算、存储、网络和应用从具体机器中抽象出来，被统一调度、弹性使用、持续交付。但当应用系统越来越互联，服务越来越开放，甚至 AI Agent 开始拥有接近自主行动能力之后，真正决定数字世界能否继续运转下去的，已经不再只是算力与架构，还有“信任”本身。

早期企业对安全有一种默认认知：安全就是拦住外面的人。服务器放在机房里，账号掌握在运维手里，系统藏在内网和防火墙后面，似乎就足够安全。这个阶段的核心信任对象是机器和位置：只要人在公司、请求来自内网、操作发生在受控服务器上，就被默认为可信。

互联网把这个默认前提真正打穿了。员工开始远程办公，业务开始开放 API，用户开始全球访问，系统也第一次从“内部可信”变成“任何人、任何服务、任何终端都可能接入”。于是过去几十年的安全演进，本质上一直在回答同一个问题：当边界不断消失，企业还能相信谁？

真正变化的从来不是某个安全产品，而是“信任对象”本身。最早企业相信机器，后来相信网络，再后来相信账号、Token 与身份系统。云时代之后，边界继续变得模糊：服务跑在云上，大数据留在本地 IDC，Kubernetes 自动调度，API 被第三方调用，CI/CD 自动部署，AI Agent 开始自主执行任务。安全世界的核心问题也随之改变：过去关注的是“谁能进入系统”，现在必须追问“谁正在代表你行动”。

AI 时代真正危险的地方也在这里。过去账号泄露，最多只是一个人被盗号；如果 Agent 拥有工具调用、数据库访问、工单处理和工作流执行能力，那么被劫持的就不只是账号，而是一个可以持续行动的数字分身。它会替你发请求、调用 API、读取知识库，甚至代表企业触发决策流程。

很多企业今天讨论 AI，仍然把注意力放在模型能力、推理速度和 GPU 成本上。但真正决定 AI 能否进入现实世界的，可能恰恰是身份体系本身。因为当 AI 开始拥有行动能力，安全问题就不再只是数据泄露，而是谁在控制行为。

从 DOS、Unix、Linux 的本地权限，到 AD、LDAP、Kerberos 的企业网络身份；从 Web 安全、防火墙、VPN，到 IAM、OAuth、Zero Trust，再到今天 AI Agent 与模型能力保护，账户与身份安全演进史，本质上就是数字世界“信任边界”不断变化的历史。

## 2.0 账户与身份安全演进史

安全史可以按技术产品切分，但更适合按“企业到底相信什么”来理解。

| 阶段 | 默认信任对象 | 代表技术 | 关键变化 |
|------|--------------|----------|----------|
| 单机时代 | 本地机器和本地管理员 | DOS、Unix、Linux 用户、文件权限、sudo | 安全边界主要在机器内部，Root 与最小权限决定本地控制权 |
| 企业网络时代 | 内网、域和组织目录 | AD、LDAP、Kerberos、域控、VPN | 身份从单机账号扩展到企业网络，登录态开始跨系统流动 |
| Web 与 API 时代 | 会话、Token 和应用身份 | Cookie、Session、OAuth、OIDC、JWT、WAF | 系统暴露到互联网，边界从机房迁移到应用入口和认证协议 |
| 云原生时代 | 工作负载、云资源和短期凭证 | IAM、IRSA、Workload Identity、SPIFFE、mTLS | 服务、Pod、Job、流水线都需要可验证、可轮换、可撤销的身份 |
| 零信任时代 | 每一次访问和每一次行为 | Zero Trust、Policy Engine、持续验证、审计 | 不再默认相信网络位置，而是按身份、设备、上下文和风险持续授权 |
| AI Agent 时代 | 代表人和组织行动的数字执行体 | Agent Identity、Tool Policy、AI Gateway、模型能力保护 | 安全边界从“访问系统”升级为“约束行为”，身份体系成为 AI 进入现实世界的前提 |

这条路径说明，安全并不是在基础设施演进之外另起一套系统。恰恰相反，越是抽象、云化、平台化和智能化，身份就越会成为新的控制面。资源可以被调度，服务可以被编排，模型可以被路由，Agent 可以被自动触发，但所有这些动作都必须回答同一个问题：这个动作是谁发起的，代表谁执行，能做什么，何时过期，失败后能不能审计和回滚。

因此，第 2 章不是把安全放在最后作为合规附录，而是把它接回《IT 基础设施演进之路》的主线：现代基础设施不只是让资源更高效，也必须让信任可表达、可验证、可撤销、可审计。

## 2.0.1 Web 攻击真正改变的不是网站，而是整个安全边界

![Web 攻击真正改变的不是网站而是整个安全边界](https://assets.onwalk.net/%E8%87%AA%E5%AA%92%E4%BD%93/%E8%BA%AB%E4%BB%BD%E4%B8%8E%E5%AE%89%E5%85%A8/2026-05-20/05-%E4%B8%BA%E4%BB%80%E4%B9%88Web%E6%94%BB%E5%87%BB%E6%94%B9%E5%8F%98%E4%BA%86%E6%95%B4%E4%B8%AA%E5%AE%89%E5%85%A8%E4%B8%96%E7%95%8C-%E5%B0%81%E9%9D%A2.png)

过去很多公司都有一种安全幻觉：只要外网防住了，内网就是安全的。

所以很长一段时间里，企业最重视的是防火墙、VPN、DMZ、专线、机房门禁。因为那时大家默认相信一件事：能进入公司网络的人，大概率就是自己人。

这种逻辑在局域网时代确实成立过。员工固定办公，服务器固定在机房，系统边界清晰，业务变化也没那么快。安全更多是在防“有人闯进来”：端口扫描、系统漏洞、病毒木马、服务器入侵。但 Web 出现后，事情彻底变了。Web 最大的改变不是“网页变多了”，而是企业第一次把业务系统直接暴露给陌生人。登录、论坛、支付、电商、后台、API，都开始面对公网用户。

攻击者不再一定要攻破整台服务器。他只需要找到一个输入框、一段脚本、一次请求、一个 Cookie、一条 API，就可能绕过权限、拿到数据，甚至进入后台。SQL 注入、XSS、CSRF、Session 劫持、Cookie 窃取，本质上都在说明一件事：过去系统默认可信的输入，突然变得不可信了。

更麻烦的是，今天的网站背后连接的早已不是简单页面，而是支付、供应链、云平台、企业后台、广告系统、用户资产，甚至 AI 服务。Web 已经从“信息入口”变成了“商业入口”，也就自然变成了最大的攻击面。于是攻击开始产业化。自动化扫描、漏洞交易、钓鱼平台、供应链投毒、AI 生成攻击脚本，让攻击门槛越来越低，而企业系统却越来越复杂。很多公司每天都在上线 API、微服务、SaaS、SDK 和第三方插件，却未必清楚自己到底暴露了多少入口。

这也是为什么传统安全边界越来越失效。过去安全靠“外网危险，内网可信”。但云计算、移动办公、多云、Kubernetes、CI/CD、远程协作，让今天的系统早就没有固定边界。服务随时创建，权限到处流动，身份跨环境访问。所以安全的核心开始从“你来自哪里”，变成“你到底是谁”。IAM、SSO、MFA、OAuth、Zero Trust、Workload Identity，本质上都是在回答同一个问题：当网络位置不再可信，系统该如何重新建立信任？

到了 AI 时代，这个问题还会继续放大。因为访问系统的不再只是人，还有 Agent、脚本、机器人、模型服务和自动化工作流。Web 攻击真正改变的，从来不只是互联网安全，而是整个数字世界的信任结构。

过去安全防的是“有人闯进来”。或许未来安全要判断的是：谁在行动？它能做什么？它为什么被允许这么做？

## 2.1 身份与访问管理 (IAM)

IAM 已经不只是账号系统，而是现代企业的安全控制平面。Keycloak、ZITADEL、Dex、Okta、Auth0 解决的不是单点登录这么简单，而是 SSO、MFA、RBAC、OIDC、SAML、LDAP 联邦，以及跨系统身份治理。

### IAM 核心概念

```
┌─────────────────────────────────────────────────────────┐
│                    IAM 核心模型                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   身份 (Identity) ──→ 认证 (Authentication) ──→ 授权   │
│       │                        │                        │
│       ▼                        ▼                        │
│   用户/服务账户            密码/Token/OAuth             │
│                                                         │
│   授权 (Authorization):                                 │
│   ┌─────────────────────────────────────────────────┐  │
│   │  Subject → Action → Resource → Condition        │  │
│   │   用户    操作      资源      条件              │  │
│   └─────────────────────────────────────────────────┘  │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 身份提供商 (IdP)

| 方案 | 类型 | 特点 |
|------|------|------|
| Keycloak | 开源 | 功能丰富，支持 OIDC/SAML |
| Okta | SaaS | 企业级，生态完善 |
| Auth0 | SaaS | 开发者友好 |
| Azure AD | 云服务 | Microsoft 生态 |

### OIDC 认证流程

```
1. 用户 → 应用: 访问资源
2. 应用 → IdP: 重定向登录
3. 用户 → IdP: 输入凭证
4. IdP → 用户: 返回授权码
5. 用户 → 应用: 授权码
6. 应用 → IdP: 换取 Token
7. IdP → 应用: Access Token + ID Token
8. 应用 → 资源服务器: 请求资源 (Bearer Token)
```

## 2.2 RBAC 与最小权限

### RBAC 模型

```
┌─────────────────────────────────────────────────────────┐
│                    RBAC 模型                            │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   用户 (User) ──┐                                       │
│                 ├──→ 角色 (Role) ──→ 权限 (Permission) │

│                           ▼                             │
│                    ┌──────────────┐                     │

│                    │  (Resource)  │                     │
│                    └──────────────┘                     │
│                                                         │
│  权限模型:                                              │
│  • resource:operation:effect                            │
│  • deployment:create:allow                              │

│                                                         │
└─────────────────────────────────────────────────────────┘
```

### Kubernetes RBAC

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: default
  name: pod-reader
rules:
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get", "list", "watch"]
---
kind: RoleBinding
metadata:
  name: read-pods
  namespace: default
subjects:
- kind: User
  name: alice
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io
```

### 最小权限原则

| 原则 | 说明 | 实践 |
|------|------|------|
| 最小权限 | 只授予必要权限 | 按需申请 |
| 权限分离 | 敏感操作多人审批 | 审批工作流 |
| 权限时效 | 临时权限，定期审查 | 临时 Token |
| 权限审计 | 定期审查权限 | 季度审计 |

## 2.3 网络安全与零信任

零信任真正改变的不是某个产品，而是默认逻辑：任何访问都不再天然可信。过去服务之间依赖内网 IP 信任；现在每个 Pod、Service、Job、Agent 都应该拥有可验证、可轮换、可撤销的工作负载身份。SPIFFE、SPIRE、mTLS 和 Service Mesh，都是把机器身份纳入安全边界的方式。

### 零信任架构

```
┌─────────────────────────────────────────────────────────┐
│                   零信任架构                             │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   传统边界模型:                          零信任模型:     │
│   ┌─────────────────┐                 ┌───────────────┐│
│   │   外网          │                 │    任何位置   ││
│   │   ┌───────────┐ │                 │    ┌────────┐ ││
│   │   │  内部网络  │ │                 │    │  验证  │ ││
│   │   │   信任    │ │                 │    │  每次  │ ││
│   │   └───────────┘ │                 │    └────────┘ ││
│   │                 │                 │       │        ││
│   └─────────────────┘                 │       ▼        ││
│                           防火墙    │    ┌────────┐    ││
│                                       │    │  授权  │    ││
│                                       │    └────────┘    ││
│                                       │       │        ││
│                                       │       ▼        ││
│                                       │    资源访问     ││
│                                       └───────────────┘│
│                                                         │
│   核心原则:                                             │
│   • 永不信任，始终验证                                   │
│   • 最小权限访问                                         │
│   • 微分段隔离                                           │
│   • 持续监控                                            │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### Kubernetes 网络策略

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: api-isolation
spec:
  podSelector:
    matchLabels:
      app: api
  policyTypes:
    - Ingress
    - Egress
  ingress:
    - from:
      - podSelector:
          matchLabels:
            app: frontend
      - podSelector:
          matchLabels:
            app: gateway
  egress:
    - to:
      - podSelector:
          matchLabels:
            app: database
    - to:
      - namespaceSelector: {}
      - podSelector:
          matchLabels:
            k8s-app: kube-dns
```

## 2.4 审计与合规

## 2.5 软件供应链安全

当身份边界继续向部署链路延伸，软件供应链安全就会成为新的上线门禁。未来不是“能跑就上线”，而是镜像要扫描、制品要签名、来源要可追溯、部署要被准入策略验证。CI/CD 身份、短期凭证、SBOM、签名和策略引擎共同决定制品是否可信。

## 2.6 身份边界取代网络边界

现代安全体系最大的变化，是边界正在迁移。传统企业安全长期围绕网络边界展开：防火墙、VPN、ACL、堡垒机、DMZ 和内外网隔离。那套模型背后的默认假设是“内网天然可信”。但云计算、移动办公、SaaS、多云、Kubernetes、CI/CD 和 AI Agent 出现后，员工不再固定在办公室，服务不再固定在机器上，应用开始跨云漂移，工作负载可能只存在几分钟。网络边界不再足以代表信任边界。

IAM 因此成为现代企业真正的控制平面。Keycloak、ZITADEL、Dex、Okta、Auth0 解决的不是简单登录，而是 SSO、MFA、RBAC、OIDC、SAML、LDAP 联邦和跨系统身份治理。企业真正危险的问题也不再只是“有没有 VPN”，而是：谁能登录，以什么角色登录，登录之后能访问什么资源。很多攻击本质上不是突破防火墙，而是拿到了一个合法身份。

零信任进一步把身份从人扩展到工作负载。现代系统中大量“访问者”并不是员工，而是 Pod、Service、Agent、CI/CD Runner、Terraform、Helm、AI Workflow 和外部 API。SPIFFE、SPIRE、mTLS、Service Mesh 要求 Service A 必须证明自己是 Service A，Service B 也必须被验证。云资源访问也开始从长期 AK/SK 转向短期身份，AWS IRSA、GCP Workload Identity、Azure Workload Identity 和 GitHub Actions OIDC 都在减少永久密钥。

最后，Secret 管理和供应链安全成为部署前门禁。Vault、KMS、External Secrets 解决的是密钥生命周期，而不是换个地方存密码；Cosign、Trivy、SBOM、SLSA、Kyverno 解决的是制品是否可信、来源是否可追溯、部署是否被策略验证。现代安全真正要回答的是：谁访问、以什么身份访问、凭证是否可信、权限是否最小化、身份是否可撤销、行为是否可审计。这才是零信任的底层逻辑。

## 本章收束

安全体系的演进说明，企业真正要治理的不是单个入口，而是一条身份链路：用户身份、工作负载身份、云资源身份、CI/CD 身份、密钥生命周期、审计日志和供应链来源。谁访问、以什么身份访问、凭证是否短期、权限是否最小、行为是否可审计，才是零信任真正的底层逻辑。

这一章也回扣第 7 章的平台工程：平台一旦成为交付入口，就必须把身份、安全、策略和审计内置到 Golden Path 中，而不是把安全留到上线前人工补票。
