# 05-04: 一图看懂 Service Mesh 兴衰

## 核心观点
Service Mesh 从热门到理性，证明了"复杂性换可观测性"的权衡。

## Service Mesh 架构

```
┌─────────────────────────────────────────────────────────────┐
│                   Service Mesh 架构                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   控制面:                数据面:                            │
│   ┌──────────┐         ┌──────────────────────────────┐   │
│   │  Istiod  │         │   Sidecar Proxy (Envoy)       │   │
│   │ • 配置   │  ──────►│   • 流量拦截                   │   │
│   │ • 发现   │         │   • 加密 (mTLS)               │   │
│   │ • 证书   │         │   • 观测                      │   │
│   └──────────┘         └──────────────────────────────┘   │
│                                                             │
│   功能: 熔断 | 限流 | 灰度 | 重试 | 超时                    │
│                                                             │
│   问题:                                                     │
│   • 复杂度高 (每个 Pod 一个 Sidecar)                        │
│   • 性能开销 (额外一跳)                                    │
│   • 运维困难 (调试困难)                                    │
│   • 收益不明 (很多功能用不上)                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```
