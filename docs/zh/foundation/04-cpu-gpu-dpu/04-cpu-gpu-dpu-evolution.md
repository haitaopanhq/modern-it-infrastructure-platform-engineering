# 01-04: 一图看懂 CPU → GPU → DPU 演进

## 核心观点

数据中心处理器正在从“CPU 负责一切”演进为“CPU、GPU、DPU 分工协作”。CPU 仍然是通用控制与复杂逻辑的中心，GPU 承担大规模并行计算，DPU/IPU 则把网络、存储、安全和虚拟化数据面从 CPU 中卸载出去。这个趋势背后的核心不是芯片名词变化，而是基础设施控制权和数据路径的重新分配。

在 AI 和云原生共同作用下，单靠 CPU 已经无法经济地处理所有任务。模型训练需要 GPU 的高并行和高带宽显存，云平台需要 DPU 在租户隔离、虚拟网络、加密、存储协议和可观测性上降低 CPU 开销。理解三者分工，是理解现代基础设施性能边界的前提。

## 图表结构

```
CPU 时代
  CPU 处理应用、系统调用、网络、存储、安全

CPU + GPU 时代
  CPU 负责控制与调度
  GPU 负责矩阵计算、渲染、并行任务

CPU + GPU + DPU 时代
  CPU 负责通用逻辑
  GPU 负责 AI 与高吞吐计算
  DPU 负责网络、存储、安全、虚拟化卸载
```

图表展示的是“谁处理数据面”的迁移。过去数据包、磁盘 I/O、加密和虚拟交换大量消耗 CPU；现在这些路径逐步被 SmartNIC、DPU、RDMA、NVMe-oF 和硬件加速接管。CPU 不是不重要，而是从所有活都干，回到更适合的控制与协调角色。

这也解释了为什么现代节点越来越像一套小型系统，而不是一块主板加几个配件。每一种处理器都带着自己的运行时、驱动、监控接口和升级节奏，平台必须把它们纳入同一条生命周期。

## 演进脉络

早期服务器的性能提升主要依赖 CPU 主频、核心数和缓存层级。云计算普及后，虚拟交换、存储协议、加密和多租户隔离让 CPU 承担越来越多基础设施开销。GPU 进入数据中心后，计算密集任务被转移到大规模并行设备，但也带来了显存、互联和调度问题。

DPU/IPU 的出现则说明基础设施自身也需要专用处理器。它让云厂商和平台团队可以把虚拟网络、存储卸载、安全策略、遥测采集和租户隔离放到主机 CPU 之外执行，既释放算力，也缩小故障域。AI 时代进一步强化了这种异构化：CPU、GPU、DPU 必须作为一个系统被设计。

## 关键技术栈

| 组件 | 典型能力 | 平台工程关注点 |
| --- | --- | --- |
| CPU | 通用计算、控制逻辑、系统调用 | 核数、NUMA、调度、隔离 |
| GPU | 矩阵计算、并行加速、显存带宽 | CUDA/ROCm、MIG、NVLink、利用率 |
| DPU/IPU | 网络、存储、安全、虚拟化卸载 | 可编程数据面、租户隔离、运维模型 |
| SmartNIC | 网络加速与队列卸载 | SR-IOV、RDMA、Telemetry |
| PCIe/CXL | 设备互联与内存扩展 | 拓扑、带宽、资源组合 |
| eBPF/DPDK/SPDK | 软件与用户态数据面 | 性能、可观测、故障隔离 |

## 误区与现实

- 误区：GPU 越多，系统越强。现实：没有 CPU、内存、网络和存储配合，GPU 会等待数据。
- 误区：DPU 只是更快网卡。现实：DPU 是基础设施处理器，目标是接管数据面和隔离面。
- 误区：异构硬件只影响底层团队。现实：调度、计费、容量、镜像和故障恢复都必须理解异构资源。

## 最佳实践

平台设计应从资源清单升级为拓扑模型。CPU、GPU、DPU、NIC、NVMe 和内存的位置关系要可发现、可调度、可观测。AI 训练节点要关注 GPU 互联和数据加载；云原生节点要关注网络卸载、存储路径和安全策略是否被硬件加速；多租户平台要明确哪些能力运行在主机 CPU，哪些运行在 DPU。

采购和架构评审不能只比较单点性能。应同时评估驱动生态、可编程能力、故障隔离、升级路径、监控接口和团队运维能力。异构计算带来的不是更多硬件标签，而是一套新的平台工程契约。

异构平台还要设计故障分层。GPU 错误可能来自驱动、显存、散热、作业框架或互联；DPU 错误可能影响整个宿主机的数据面；PCIe 或 CXL 问题可能表现为随机 I/O 抖动。平台不能只暴露“设备健康”一个布尔值，而要把错误来源、隔离范围和自动恢复动作记录清楚。

对业务而言，最好的体验不是理解每一种芯片，而是申请到适合任务的能力包。例如“低延迟推理节点”“高吞吐训练节点”“安全隔离网关节点”。平台内部可以很复杂，但对外应形成稳定的资源画像和服务等级。
