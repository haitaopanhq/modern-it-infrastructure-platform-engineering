# 01-05: 一图看懂从 BIOS 到 Kubernetes

## 核心观点

从 BIOS 到 Kubernetes 的历史，本质上是基础设施控制权不断上移、再被重新组织的历史。最早控制权掌握在硬件固件和系统管理员手中，后来 Hypervisor 把物理机变成虚拟机资源池，云平台把资源交付变成 API，Kubernetes 又把应用运行、调度和服务治理抽象成声明式对象。

但控制权上移不等于底层消失。BIOS/UEFI、BMC、PXE、Linux、Hypervisor、云 API 和 Kubernetes 仍然串成一条启动与交付链。平台工程的任务，是把这条链变成稳定、可观测、可审计的产品能力，而不是让每一层各自为政。

## 图表结构

```
BIOS/UEFI
  -> 硬件初始化、启动顺序、固件策略
BMC/PXE
  -> 带外管理、自动装机、故障恢复
Linux/Hypervisor
  -> OS 能力、虚拟化、资源隔离
Cloud API
  -> 资源池、计费、权限、多租户
Kubernetes
  -> 容器编排、服务发现、声明式交付
Platform Engineering
  -> 自助服务、策略、Golden Path、开发者体验
```

这张图不是线性淘汰关系，而是控制层叠加关系。Kubernetes 位于上层，但它依赖节点启动、镜像分发、网络插件、存储插件、内核能力和云控制器。任一层失真，上层体验都会变成不可靠的“黑盒”。

读图时可以把每一层看成一个控制面：固件控制启动，带外系统控制恢复，操作系统控制进程和设备，云平台控制资源池，Kubernetes 控制应用副本，IDP 控制开发者路径。平台工程要做的是让这些控制面协同，而不是互相覆盖。

## 演进脉络

物理机时代，交付一台服务器需要采购、上架、装系统和手工配置，控制权靠流程与脚本维持。虚拟化时代，Hypervisor 提供了更高的利用率和更快的虚拟机生命周期管理。云时代，API、IAM、计费和区域化资源池成为新的控制中心。

容器与 Kubernetes 把应用交付进一步标准化，开发者可以通过 YAML 描述期望状态，平台负责调度、重启、服务发现和扩缩容。进入平台工程阶段，组织关注点从“有没有 Kubernetes”转向“开发者能否通过可信路径交付软件”。AI 与 Agent 又让控制面出现新入口：任务可能由人、流水线或智能代理共同触发。

## 关键技术栈

| 层级 | 代表技术 | 平台工程关注点 |
| --- | --- | --- |
| 固件与带外 | BIOS/UEFI, BMC, IPMI/Redfish | 启动、远程恢复、资产可信 |
| 装机与镜像 | PXE, iPXE, Packer, cloud-init | 标准化基线与可重复交付 |
| 操作系统 | Linux, systemd, kernel modules | 节点生命周期与运行时依赖 |
| 虚拟化 | KVM, QEMU, ESXi, libvirt | 资源池、多租户、隔离 |
| 云控制面 | OpenStack, 公有云 API | IAM、网络、计费、弹性 |
| Kubernetes | kubelet, CRI, CNI, CSI, Operator | 声明式控制与扩展 |

## 误区与现实

- 误区：Kubernetes 是基础设施终点。现实：它是重要控制层，但仍依赖硬件、OS、网络、存储和云控制面。
- 误区：平台工程只是给 Kubernetes 做门户。现实：平台要负责从资源申请到交付、观测、合规和成本的完整路径。
- 误区：自动化越多越好。现实：没有清晰权责和状态来源的自动化，只会把故障传播得更快。

## 最佳实践

平台团队应为每一层定义事实源：资产与固件状态来自带外系统，节点基线来自镜像与配置管理，运行状态来自 Kubernetes 与可观测系统，权限与成本来自云控制面或组织平台。不要让多个系统同时声明同一资源状态。

建设平台时，先画清楚启动链、控制链和故障链。一次应用发布背后涉及镜像、节点、网络、存储、证书、策略和监控，任何一个环节都需要可回放、可追踪。平台工程的成熟度，不是 UI 有多漂亮，而是控制权迁移后，团队是否仍能解释系统为什么这样运行。

实践中可以把控制链拆成三类：资源控制、运行控制和体验控制。资源控制回答机器、网络、存储和加速卡从哪里来；运行控制回答应用如何启动、扩缩、恢复和观测；体验控制回答开发者如何自助完成交付。三类控制如果混在一个脚本或门户里，平台会越来越难治理。

AI 与 Agent 场景还会放大这个问题。未来任务可能由自动化代理提交，但代理不能绕过固件、镜像、权限、配额和审计。越是自动化，越需要清晰的控制边界和事实源，否则只是把人工误操作变成机器速度的误操作。
