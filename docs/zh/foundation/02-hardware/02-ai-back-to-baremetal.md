# 01-02: 一图看懂 AI 为什么重新回到裸金属

## 核心观点

AI 工作负载把基础设施从“通用计算抽象”重新拉回“硬件拓扑约束”。传统 Web 服务通常关注请求吞吐、水平扩容和无状态副本，虚拟机或容器带来的抽象成本可以被弹性和工程效率抵消。大模型训练、推理和多模态处理则不同，瓶颈集中在 GPU 算力、显存容量、显存带宽、GPU 间通信、RDMA 网络和数据加载链路上。

因此，AI 并不是简单地否定云计算，而是让裸金属重新成为关键选项。裸金属的价值在于减少不必要的虚拟化层，稳定暴露 GPU、NIC、NVMe 与 NUMA 拓扑，让调度器和平台团队能围绕真实硬件做放置、隔离和容量管理。

## 图表结构

```
Web 工作负载
  -> HTTP 请求 / 无状态副本 / 负载均衡
  -> CPU 与网络吞吐为主
  -> 虚拟机、容器、Serverless 均可承载

AI 工作负载
  -> 训练 / 推理 / 向量检索 / 多模态处理
  -> GPU、HBM、NVLink、RDMA、NVMe 共同决定上限
  -> 更依赖裸金属或接近裸金属的资源交付
```

图中对比的是扩展逻辑：Web 扩容常常是增加副本，AI 扩容则要回答 GPU 是否同代、是否同机、是否同交换域、通信是否走 RDMA、数据是否能及时喂满算力。一个 GPU 节点的价值不只在卡数，还在卡、CPU、内存、网卡和存储之间的拓扑关系。

因此，AI 平台的“资源规格”应该比传统云主机更具体。除了卡数，还要说明显存、互联、网络域、本地盘、镜像栈和可用调度队列。否则用户看到的是统一入口，实际得到的是差异很大的硬件路径。

## 演进脉络

虚拟化和云平台把通用服务器变成可编程资源池，解决了交付速度和利用率问题。容器与 Kubernetes 进一步把应用发布和弹性伸缩标准化，使大多数互联网服务可以不关心物理机细节。

AI 时代的变化在于单个任务重新变重。训练任务需要长时间占用成组 GPU，推理任务需要在成本、延迟和吞吐之间精细平衡，数据预处理与 checkpoint 写入又会冲击网络和存储。抽象层如果不能准确表达 GPU 拓扑、MIG、RDMA、NVLink 和本地 NVMe，就会把性能问题隐藏成“平台不稳定”。

## 关键技术栈

| 技术 | 作用 | 关键关注点 |
| --- | --- | --- |
| CUDA/ROCm | GPU 编程与运行时 | 驱动、库、框架版本一致性 |
| NCCL/RCCL | 多 GPU 通信 | 拓扑感知、网络插件、超时诊断 |
| NVLink/PCIe | GPU 与主机互联 | 带宽、代际、跨 NUMA 影响 |
| RDMA/RoCE/InfiniBand | 跨节点通信 | PFC/ECN、GID、MTU、队列深度 |
| GPUDirect Storage/RDMA | 减少 CPU 中转 | 数据路径与权限配置 |
| Kubernetes Device Plugin | GPU 资源暴露 | 卡粒度、MIG、健康检查 |

## 误区与现实

- 误区：AI 只需要更多 GPU。现实：GPU 如果被慢网络、慢存储或错误拓扑限制，利用率会很低。
- 误区：裸金属意味着放弃平台化。现实：裸金属也可以通过 PXE、BMC、Kubernetes、调度器和自动化交付成平台能力。
- 误区：虚拟化一定不可用。现实：开发、轻量推理和隔离需求仍可使用虚拟化；关键在于它是否破坏性能路径。

## 最佳实践

AI 集群要先建立硬件画像，再谈平台抽象。节点应记录 GPU 型号、显存、NVLink 拓扑、PCIe 插槽、CPU NUMA、网卡速率、本地盘能力和驱动版本。调度层要支持拓扑亲和，避免把强通信任务分散到高延迟路径上。

交付上可以保持云原生接口，但资源模型不能只剩 `gpu: 8`。训练平台应提供节点组、网络域、存储路径、镜像基线和健康检查；推理平台应把批处理、KV Cache、显存碎片和冷启动成本纳入容量模型。裸金属不是倒退，而是让平台对真实瓶颈负责。

资源交付还应区分训练、推理、开发和批处理。训练强调成组 GPU、长时间稳定运行和高速互联；推理强调显存利用率、延迟分位数和弹性扩缩；开发环境强调快速分配与成本回收；批处理强调队列、公平性和可抢占策略。不同场景混用同一资源池，往往会让高价值 GPU 被低优先级任务占住。

成本治理也要回到硬件事实。AI 集群最贵的不是服务器外壳，而是 GPU 空闲、网络设计失误、数据路径过慢和调度不感知拓扑造成的浪费。平台如果能让每个作业知道自己为什么被放在某组节点上，才算真正把裸金属能力产品化。
