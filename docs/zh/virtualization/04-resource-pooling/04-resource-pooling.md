# 02-04: 一图看懂资源池化历史

## 核心观点

资源池化是云计算和平台工程的基础动作：把分散的服务器、磁盘、网络和加速卡，从“某台机器的固定资产”变成“可被调度、计量、隔离和回收的共享资源”。它让组织可以提升利用率、缩短交付周期，并用 API 管理容量。但资源池化不是把所有资源混在一起，真正难点在于如何保留性能、故障域和安全边界。

从虚拟机资源池到超融合，再到 Kubernetes 集群、多云和 AI 算力池，池化对象不断扩展。计算、存储、网络可以池化，GPU、IP、证书、域名、队列、数据库实例也可以池化。平台工程的价值，是把这些池化能力变成稳定的自助服务。

## 图表结构

```
单机资源
  -> 固定服务器、固定磁盘、固定网络端口
虚拟化资源池
  -> VM、共享存储、虚拟交换
云资源池
  -> Region / AZ / VPC / EBS / LB
云原生资源池
  -> Kubernetes Node / Pod / PVC / Service
AI 与异构资源池
  -> GPU Pool / RDMA Fabric / 本地 NVMe / 模型服务
```

图表表达的是资源所有权的变化。单机时代资源归某台机器；虚拟化时代资源归集群；云时代资源归控制面；平台工程时代资源归产品化能力。用户不直接占有机器，而是申请符合策略的资源份额。

池化之后，资源从“买回来就归某个项目”变成“按策略流动”。这要求平台既能快速分配，也能及时回收，还能解释为什么某类资源不能随意共享。没有这些规则，池化很快会退化成新的手工分配。

## 演进脉络

早期资源池化主要围绕计算虚拟化展开，VMware DRS、KVM 集群和共享存储让虚拟机可以在主机之间调度和迁移。随后存储、网络也被纳入统一池，形成云平台的基础能力。OpenStack 和公有云把这些能力包装成租户、配额、镜像、网络和计费接口。

超融合把计算与存储合并到同一节点池，降低小规模环境复杂度。Kubernetes 则把应用运行所需的 CPU、内存、网络、存储和扩展资源抽象为声明式对象。AI 时代的池化更复杂：GPU 不是简单数量，卡型、拓扑、互联、显存和租户隔离共同决定资源能否共享。

## 关键技术栈

| 池化对象 | 代表技术 | 关键能力 |
| --- | --- | --- |
| 计算 | vSphere DRS, KVM, Kubernetes Scheduler | 调度、隔离、迁移 |
| 存储 | Ceph, vSAN, CSI, S3 | 容量、复制、快照 |
| 网络 | VXLAN, SDN, CNI, LB | 隔离、路由、入口 |
| 安全 | IAM, RBAC, OPA, NetworkPolicy | 权限、策略、审计 |
| 异构算力 | Device Plugin, Volcano, Slurm, Ray | GPU 调度、队列、公平性 |
| 平台入口 | Backstage, Port, 自研 IDP | 自助申请与生命周期 |

## 误区与现实

- 误区：池化越大越好。现实：池越大，调度效率可能更高，但故障半径、网络复杂度和治理成本也会上升。
- 误区：资源池可以隐藏所有差异。现实：卡型、磁盘介质、网络域和地域差异必须被模型化，否则会变成隐性故障。
- 误区：有配额就等于治理。现实：治理还需要审批、成本归属、回收、优先级和容量预测。

## 最佳实践

设计资源池要先定义池边界：按地域、可用区、硬件代际、租户安全级别、网络域和工作负载类型划分。对通用计算可以追求较大共享池，对数据库、低延迟网关、AI 训练等场景应保留专用池或带拓扑约束的池。

平台层要提供资源生命周期管理，包括申请、审批、交付、扩容、缩容、监控、告警、成本归属和自动回收。容量治理要结合趋势预测和实际利用率，避免一边资源闲置，一边关键业务排队。资源池化的目标不是平均主义，而是让资源在正确边界内高效流动。

资源池还需要明确准入和退出机制。哪些工作负载可以进入共享池，哪些必须使用专用池，哪些资源到期后自动回收，哪些资源需要保留审计证据，都应由平台策略表达。没有退出机制的资源池，最终会变成无人敢动的历史包袱。

对 AI 和高性能场景，资源池化更要避免“只按数量分配”。同样 8 张 GPU，可能因为拓扑、网络域、显存、驱动和作业队列不同而价值完全不同。平台应把这些差异变成调度标签、资源等级和服务说明，让用户选择能力，而不是猜测机器。
