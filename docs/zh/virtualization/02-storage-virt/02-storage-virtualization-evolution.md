# 02-02: 一图看懂存储虚拟化演进

## 核心观点

存储虚拟化的目标，是把分散、异构、容量有限的磁盘资源，抽象成可按需分配、可复制、可扩展、可迁移的数据服务。它从 RAID 和阵列开始，经过 SAN/NAS 的集中化，再到 Ceph、vSAN、MinIO、对象存储和 CSI 驱动，把“磁盘”变成了块、文件、对象和卷声明。

很多人以为存储只是“硬盘”。但如果真正回看过去二十年的基础设施演进，会发现现代存储的发展，本质上也是一场资源抽象史。最早的本地磁盘时代，服务器直接挂载 HDD，依赖 RAID、ext4、XFS 管理数据。那个阶段，存储几乎完全绑定物理机器，扩容困难、迁移复杂、利用率低。后来企业进入 IDC 时代，SAN/NAS、Fibre Channel、iSCSI、NFS 和存储阵列开始普及。存储第一次从单台服务器里抽离出来，变成机房里的集中式基础设施。企业开始拥有统一存储、集中备份、高可用、快照、容灾和远程挂载能力。

与此同时，LVM、Thin Provision、Snapshot 开始出现，磁盘从“物理设备”变成“逻辑卷”。真正重要的变化，不是容量变大，而是抽象层变厚。运维终于可以动态扩容、快照、迁移与按需分配资源。互联网规模继续扩大后，HDFS、GlusterFS、Ceph 开始出现，存储正式进入分布式时代。对象存储、块存储、文件存储开始统一进入软件定义体系，Ceph 本质上做的，就是把离散磁盘抽象成统一存储池。

云计算兴起后，存储进一步变成服务。云硬盘、对象存储、文件存储、归档存储、备份服务、跨区域复制，让企业不再直接购买存储阵列，而是按容量、性能、可用性和生命周期消费存储能力。云原生时代之后，CSI、Rook、Longhorn、OpenEBS 又继续向上抽象，存储开始与 Kubernetes 生命周期绑定，真正进入“可编排时代”。

AI 时代则重新改变了存储目标。传统数据库时代拼的是 IOPS，而 AI Storage 开始拼吞吐、数据流与 GPU Feeding。NVMe-oF、GPUDirect Storage、Checkpoint，本质都在解决同一个问题：GPU 不能等待 IO。AI 时代，昂贵的不只是 GPU，还有计算等待 IO。

但存储虚拟化不是把硬盘藏起来这么简单。它必须同时处理一致性、延迟、吞吐、故障恢复、数据放置、快照、备份、配额和成本。对平台工程来说，存储是最容易被上层抽象误解的资源：创建一个 PVC 很简单，保证它在故障、扩容、迁移和高并发下仍然可信，才是难点。

## 图表结构

```
本地磁盘 / RAID
  -> 单机可靠性与容量聚合
SAN / NAS
  -> 网络集中存储，块与文件协议
软件定义存储
  -> Ceph / vSAN / Gluster / MinIO
云存储与 Kubernetes
  -> EBS / OSS / S3 / CSI / Snapshot
AI 与数据密集场景
  -> NVMe-oF / 本地 NVMe / 并行文件系统
```

这张图按照“数据离计算多远”来理解更清楚。本地盘延迟低但管理复杂；集中存储便于运维但依赖网络；对象存储扩展性强但不适合所有 POSIX 场景；AI 训练又把本地高速盘、并行文件系统和对象存储预热组合起来。

存储虚拟化还要同时回答“数据在哪里”和“谁对数据负责”。一个卷可能挂在 Pod 上，但副本、快照、备份、加密、跨区复制和删除保护都在平台侧完成。用户看到容量，平台要管理生命周期。

## 演进脉络

RAID 时代关注单机内的可靠性和容量，SAN/NAS 时代把存储从服务器中抽离出来，形成集中管理的企业存储。虚拟化普及后，虚拟机磁盘、共享数据存储和快照能力成为云平台基础。软件定义存储进一步把普通服务器磁盘聚合成分布式存储池，用副本、纠删码和一致性协议提升可用性。

云原生时代，Kubernetes 通过 CSI 把卷生命周期交给存储插件，让应用可以声明所需容量、访问模式和存储类。AI 与大数据场景则提出更高吞吐要求：训练数据集、checkpoint、向量索引和模型权重需要在对象存储、本地 NVMe、缓存层和高速网络之间流动。

## 关键技术栈

| 类型 | 代表技术 | 适用场景 |
| --- | --- | --- |
| 块存储 | iSCSI, FC, RBD, EBS, NVMe-oF | 数据库、虚拟机磁盘 |
| 文件存储 | NFS, SMB, CephFS, Lustre | 共享目录、训练数据 |
| 对象存储 | S3, Swift, MinIO, OSS | 非结构化数据、归档、模型文件 |
| SDS | Ceph, vSAN, Longhorn | 云平台与边缘资源池 |
| Kubernetes 存储 | CSI, StorageClass, VolumeSnapshot | 声明式卷管理 |
| 数据保护 | Snapshot, Backup, Replication | 恢复点与跨域容灾 |

## 误区与现实

- 误区：对象存储可以替代所有存储。现实：对象存储适合海量非结构化数据，但不等同于低延迟块设备或 POSIX 文件系统。
- 误区：三副本就是高可用。现实：副本策略还要考虑故障域、网络分区、恢复速度和一致性窗口。
- 误区：PVC 创建成功就代表存储可用。现实：真正风险常出现在扩容、节点故障、快照恢复和高并发 I/O。

## 最佳实践

平台应把存储类设计成产品，而不是把底层参数直接丢给业务。不同工作负载应有清晰的性能、可用性、备份和成本等级，例如数据库块存储、共享文件存储、对象归档、AI 高吞吐缓存和临时本地盘。

上线前要验证故障域、恢复时间、快照一致性和扩容路径。监控上除了容量，还要看 IOPS、吞吐、延迟分位数、队列深度、重平衡状态和错误率。对 AI 场景，要把数据预热、checkpoint 写入和对象存储带宽纳入训练作业设计，避免 GPU 等数据。

应用侧也需要理解存储语义。块存储适合数据库，但并不自动解决备份；共享文件系统方便协作，但元数据热点可能成为瓶颈；对象存储扩展性强，但读写模型与本地文件系统不同。平台应通过文档、模板和默认策略帮助团队选对类型，而不是让所有人都申请同一种卷。

存储变更要格外谨慎。扩容、迁移、重平衡、版本升级和故障恢复都可能影响尾延迟。成熟平台会把这些动作纳入维护窗口、风险评估和回滚方案，并在业务层明确恢复点目标和恢复时间目标。
