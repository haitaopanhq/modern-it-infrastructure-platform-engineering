# 02-01: 一图看懂计算虚拟化演进

## 核心观点

计算虚拟化的主线，是把物理 CPU、内存和设备抽象成可分配、可迁移、可隔离的计算单元。它不是单一技术，而是一组围绕隔离、性能和管理效率不断权衡的方案：软件模拟提供兼容性，半虚拟化减少陷入开销，硬件辅助虚拟化成为云平台主流，容器则把操作系统级隔离推到应用交付层。

今天的基础设施并没有在虚拟机和容器之间二选一。公有云、私有云、边缘节点、AI 集群和安全沙箱会同时使用 KVM、QEMU、ESXi、Firecracker、Kata Containers、containerd 等技术。关键问题不是“谁取代谁”，而是哪一层负责隔离，哪一层负责调度，哪一层承担性能成本。

很多人以为，容器已经淘汰了虚拟化。但如果真正回看过去二十多年的基础设施演进，会发现虚拟化从未消失，它只是不断向下沉，越来越靠近硬件。容器、Kubernetes 和 AI Infra 不是虚拟化的反面，而是把资源抽象、隔离和调度继续推向更高层的结果。

## 图表结构

```
软件模拟
  -> QEMU TCG / Bochs
  -> 兼容性强，性能成本高

半虚拟化
  -> Xen PV / virtio 思路
  -> Guest 与 Host 协作降低开销

硬件辅助全虚拟化
  -> KVM / ESXi / Hyper-V
  -> VT-x / AMD-V / IOMMU 提供隔离与性能

OS 级虚拟化
  -> containerd / runc / Docker
  -> 共享内核，启动快，隔离边界不同
```

这张图的核心是隔离边界的变化。传统虚拟机隔离在硬件抽象层，每个 Guest OS 拥有自己的内核；容器隔离在 Linux 内核对象层，多个容器共享宿主机内核；MicroVM 和安全容器则试图在两者之间取得平衡。

因此，评价计算虚拟化不能只看启动速度。更关键的是隔离边界在哪里、谁负责补丁、设备如何暴露、故障如何止血、资源如何回收。不同边界适合不同风险模型，混用并不可怕，边界不清才可怕。

## 演进脉络

最早的服务器时代，应用直接运行在物理机上。一台服务器往往只承载少数业务，CPU、内存、存储和网络资源很难被充分利用，扩容也更多依赖继续采购物理服务器。业务增长之后，真正暴露出来的问题不是“机器不够”，而是资源被锁死在一台台机器里，无法被统一池化、切分和调度。

早期软件模拟解决的是“让一个平台运行另一个平台”的问题，适合兼容、开发和架构验证，但性能不足以支撑大规模生产。半虚拟化通过修改或适配 Guest，让 Guest 主动配合 Hypervisor，减少敏感指令处理和 I/O 模拟开销。

真正改变行业的，是 CPU 厂商后来加入硬件辅助虚拟化能力。Intel VT-x、AMD-V 出现后，CPU 开始原生支持虚拟化。过去 Hypervisor 需要大量 Trap & Emulate，很多 privileged instructions 很难高效处理，性能与隔离成本都很高；硬件辅助虚拟化本质上是在 CPU 内新增虚拟化执行模式，让 Guest OS 能以更低开销运行。

后来 EPT/NPT、IOMMU、VT-d、SR-IOV、PCIe Passthrough 和 vGPU 继续成熟，行业开始从软件模拟为主，进入硬件辅助虚拟化时代。KVM、VMware ESXi、Xen、Hyper-V 等方案因此真正成为数据中心基础能力。服务器第一次被系统性抽象成资源池：CPU、内存、网络、存储都可以被切分、隔离、调度与迁移。

虚拟化真正改变的，并不只是单机性能，而是资源利用率、弹性能力、故障隔离与交付方式。它让服务器从固定资产变成可分配资源，也让云控制面有了可以统一管理的底座。

后来 Docker 与 Kubernetes 出现，行业进入容器化与云原生编排阶段。VM 虚拟的是硬件环境，Container 隔离的是进程运行环境。Linux namespaces、cgroups、本地文件系统隔离与容器镜像，共同构成了容器的核心能力。Kubernetes 则进一步把容器运行环境纳入声明式 API、调度器、控制循环、服务发现、网络与存储插件体系之中。

很多人以为容器在消灭 VM，但现实世界里，大量 Kubernetes 集群依然运行在 VM 之上。容器提升应用交付密度和速度，虚拟机仍承担租户边界、宿主隔离、异构系统和云平台资源池的职责。近年的 Firecracker、Cloud Hypervisor、Kata Containers 又把轻量 VM 带回云原生语境，说明虚拟化不是退出舞台，而是在新的隔离需求下重新组合。

AI Infra 出现后，行业又重新贴近硬件。GPU Cluster 真正拼的，已经不只是 CPU 调度，而是 PCIe 拓扑、NUMA 亲和性、RDMA、NVLink、显存带宽、GPU Direct 与 NCCL 通信效率。于是 GPU Passthrough、SR-IOV、vGPU、DPU、SmartNIC 再次成为关键能力，本质上都在解决同一个问题：如何在保留资源抽象、隔离和调度能力的同时，尽可能降低虚拟化损耗。

所以现代基础设施真正的演进，并不是虚拟化被淘汰，而是物理机、硬件辅助虚拟化、VM、容器化、Kubernetes 和 AI Infra 沿着同一条资源抽象链条继续前进。整个行业一直都在持续强化资源抽象能力，只是越到 AI 时代，抽象越必须尊重硬件真实边界。

## 关键技术栈

| 技术 | 作用 | 平台工程关注点 |
| --- | --- | --- |
| KVM | Linux 内核虚拟化模块 | 云主机、私有云、宿主机能力 |
| QEMU | 设备模拟与虚拟机进程 | virtio、镜像、设备模型 |
| libvirt | 虚拟机管理 API | 生命周期、网络、存储编排 |
| VT-x/AMD-V | CPU 硬件辅助虚拟化 | 性能、隔离、兼容性 |
| IOMMU/SR-IOV | 设备隔离与直通 | 网卡/GPU 性能与安全 |
| OCI Runtime | 容器运行标准 | 镜像、namespace、cgroups |
| Firecracker/Kata | MicroVM 与安全容器 | 多租户 Serverless 与沙箱 |

## 误区与现实

- 误区：容器已经淘汰虚拟机。现实：容器提升交付效率，虚拟机仍承担强隔离、多租户和异构 OS 场景。
- 误区：虚拟化只带来性能损耗。现实：虚拟化同时带来资源池化、迁移、快照、隔离和运维效率。
- 误区：KVM 只是 Linux 上的一个功能。现实：KVM 加 QEMU、网络、存储、镜像和控制面才构成完整虚拟化平台。

## 最佳实践

选择计算虚拟化方案时，应先明确隔离强度、启动速度、设备需求和运维边界。生产多租户云主机通常优先硬件辅助虚拟化；CI、边缘和 Serverless 可评估 MicroVM；常规应用交付适合容器；强安全容器可以使用 Kata 或 gVisor 类方案。

平台层要统一镜像、网络、存储和观测接口，避免虚拟机与容器形成两套孤岛。对于 GPU、RDMA 和高性能网卡场景，要提前验证 IOMMU、SR-IOV、设备插件和迁移限制。虚拟化的成熟度，体现在资源生命周期是否可自动化，而不只是能否启动一个 VM。

架构上建议把虚拟机、容器和 MicroVM 看作不同隔离边界，而不是互相排斥的产品线。虚拟机适合租户边界和异构系统，容器适合应用交付和快速扩缩，MicroVM 适合不可信代码、Serverless 与短生命周期任务。统一平台可以对外提供同样的申请、审计和观测体验，对内选择不同运行底座。

此外，虚拟化方案必须配合组织能力。没有宿主机补丁、镜像治理、网络排障和容量管理，任何先进 Hypervisor 都会变成风险源。平台设计要从第一天就考虑升级、迁移、回滚和故障隔离，而不是只验证 happy path。
