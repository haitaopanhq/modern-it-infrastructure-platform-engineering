# 02-01: 一图看懂计算虚拟化演进

## 核心观点

计算虚拟化的主线，是把物理 CPU、内存和设备抽象成可分配、可迁移、可隔离的计算单元。它不是单一技术，而是一组围绕隔离、性能和管理效率不断权衡的方案：软件模拟提供兼容性，半虚拟化减少陷入开销，硬件辅助虚拟化成为云平台主流，容器则把操作系统级隔离推到应用交付层。

今天的基础设施并没有在虚拟机和容器之间二选一。公有云、私有云、边缘节点、AI 集群和安全沙箱会同时使用 KVM、QEMU、ESXi、Firecracker、Kata Containers、containerd 等技术。关键问题不是“谁取代谁”，而是哪一层负责隔离，哪一层负责调度，哪一层承担性能成本。

## 图表结构

```
软件模拟
  -> QEMU TCG / Bochs
  -> 兼容性强，性能成本高

半虚拟化
  -> Xen PV / virtio 思路
  -> Guest 与 Host 协作降低开销

硬件辅助全虚拟化
  -> KVM / ESXi / Hyper-V
  -> VT-x / AMD-V / IOMMU 提供隔离与性能

OS 级虚拟化
  -> containerd / runc / Docker
  -> 共享内核，启动快，隔离边界不同
```

这张图的核心是隔离边界的变化。传统虚拟机隔离在硬件抽象层，每个 Guest OS 拥有自己的内核；容器隔离在 Linux 内核对象层，多个容器共享宿主机内核；MicroVM 和安全容器则试图在两者之间取得平衡。

因此，评价计算虚拟化不能只看启动速度。更关键的是隔离边界在哪里、谁负责补丁、设备如何暴露、故障如何止血、资源如何回收。不同边界适合不同风险模型，混用并不可怕，边界不清才可怕。

## 演进脉络

早期软件模拟解决的是“让一个平台运行另一个平台”的问题，适合兼容、开发和架构验证，但性能不足以支撑大规模生产。半虚拟化通过修改或适配 Guest，让 Guest 主动配合 Hypervisor，减少敏感指令处理和 I/O 模拟开销。

硬件辅助虚拟化成熟后，KVM、ESXi、Hyper-V 等方案成为云计算的底座。CPU 虚拟化、内存页表扩展、IOMMU 和 virtio 设备模型让虚拟机兼具隔离与性能。容器兴起后，应用交付速度进一步提升，但它没有替代虚拟机的安全与多租户边界。近年的 Firecracker、Cloud Hypervisor、Kata Containers 又把轻量 VM 带回云原生语境。

## 关键技术栈

| 技术 | 作用 | 平台工程关注点 |
| --- | --- | --- |
| KVM | Linux 内核虚拟化模块 | 云主机、私有云、宿主机能力 |
| QEMU | 设备模拟与虚拟机进程 | virtio、镜像、设备模型 |
| libvirt | 虚拟机管理 API | 生命周期、网络、存储编排 |
| VT-x/AMD-V | CPU 硬件辅助虚拟化 | 性能、隔离、兼容性 |
| IOMMU/SR-IOV | 设备隔离与直通 | 网卡/GPU 性能与安全 |
| OCI Runtime | 容器运行标准 | 镜像、namespace、cgroups |
| Firecracker/Kata | MicroVM 与安全容器 | 多租户 Serverless 与沙箱 |

## 误区与现实

- 误区：容器已经淘汰虚拟机。现实：容器提升交付效率，虚拟机仍承担强隔离、多租户和异构 OS 场景。
- 误区：虚拟化只带来性能损耗。现实：虚拟化同时带来资源池化、迁移、快照、隔离和运维效率。
- 误区：KVM 只是 Linux 上的一个功能。现实：KVM 加 QEMU、网络、存储、镜像和控制面才构成完整虚拟化平台。

## 最佳实践

选择计算虚拟化方案时，应先明确隔离强度、启动速度、设备需求和运维边界。生产多租户云主机通常优先硬件辅助虚拟化；CI、边缘和 Serverless 可评估 MicroVM；常规应用交付适合容器；强安全容器可以使用 Kata 或 gVisor 类方案。

平台层要统一镜像、网络、存储和观测接口，避免虚拟机与容器形成两套孤岛。对于 GPU、RDMA 和高性能网卡场景，要提前验证 IOMMU、SR-IOV、设备插件和迁移限制。虚拟化的成熟度，体现在资源生命周期是否可自动化，而不只是能否启动一个 VM。

架构上建议把虚拟机、容器和 MicroVM 看作不同隔离边界，而不是互相排斥的产品线。虚拟机适合租户边界和异构系统，容器适合应用交付和快速扩缩，MicroVM 适合不可信代码、Serverless 与短生命周期任务。统一平台可以对外提供同样的申请、审计和观测体验，对内选择不同运行底座。

此外，虚拟化方案必须配合组织能力。没有宿主机补丁、镜像治理、网络排障和容量管理，任何先进 Hypervisor 都会变成风险源。平台设计要从第一天就考虑升级、迁移、回滚和故障隔离，而不是只验证 happy path。
