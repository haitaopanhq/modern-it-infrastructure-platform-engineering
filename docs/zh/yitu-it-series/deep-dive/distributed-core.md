# 分布式系统核心理论：CAP / ZooKeeper / Raft / HA

> **分布式系统的本质不是把机器变多，而是把不确定性关进笼子里**

## 核心判断

现代基础设施真正难的不是"多部署几台机器"，而是让一群随时可能失败的机器，在网络不可靠、磁盘不稳定、进程会崩溃的世界里，依然对外表现得像一个可靠系统。

## CAP 理论

CAP：Consistency（一致性）、Availability（可用性）、Partition tolerance（分区容错性）。

在网络分区发生时，系统必须做选择：C 还是 A。互联网系统、缓存系统、消息系统、数据库集群，最后都会在这里做取舍。

## ZooKeeper

ZooKeeper 回答的是"在分布式系统里我该相信谁"的问题。通过 ZAB 协议让多个节点形成一致状态。Hadoop、HBase、Kafka 早期架构都依赖 ZooKeeper。

自建 ZooKeeper 要求团队真正理解 quorum、多数派、脑裂、磁盘延迟。

## Raft

Raft 是对共识算法的一次工程化简化。etcd、Consul、TiKV、CockroachDB 都采用 Raft。Raft 和 ZK 的关系：ZK 更像早期分布式协调中心，Raft 更像后来被广泛嵌入各类系统内部的共识内核。

## HA（高可用）

业务最终关心的不是 ZK 还是 Raft，而是系统能不能少宕机。HA 从业务角度看是连续性，从系统角度看是故障检测、主备切换、状态复制、流量摘除、恢复编排。

## 从开源到云服务

- 自建：控制力强、成本可控，但运维极重
- 云服务：省心，但成本高、厂商绑定
- 核心判断：如果它决定了集群生死、流量调度、数据库主从、服务注册，你就必须理解它——哪怕用云服务，也不能把它当黑盒
