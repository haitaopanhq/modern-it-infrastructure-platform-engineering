本目录用于沉淀公众号「行者深蓝」的一图类文章，并把它们改造成一本可以持续扩写的开源电子书。

第一大系列《现代 IT 基础设施演进史》是长期置顶的总宇宙。它不按工具分类，也不写成教程，而是用技术史的方式回答一个长期问题：当业务增长、系统复杂度、成本压力和安全边界不断变化时，企业基础设施为什么会从单机、网络、虚拟化、云平台、云原生一路走向 AI Native。

详细章节表见 [book-outline.md](./book-outline.md)。

开篇引导：[为什么写《现代 IT 基础设施演进史》](./00-series-review/00-preface-why-this-series.md)。这篇放在技术目录之前，说明这个系列不是工具教程，而是一个长期从事基础设施工作的人，在行业变化、职业低谷和 AI 时代重新整理经验、对抗复杂度失控的一次记录。

开篇简史的第一阶段收尾篇是 [AI 会重新定义下一代基础设施吗？](./essays/ai-redefine-infra.md)。这篇文章把“业务增长 -> 资源失衡 -> 虚拟化 -> 云平台弹性 -> 公有云崛起 -> 私有云未消失 -> 混合云 -> 多云 / 云中立 -> 平台工程 -> AI Infra”收束为一条控制权迁移线，并把后续 AI Native、Agent Runtime、AI Gateway 和多 Agent 治理接入全书主线。

## 三层叙事结构

这套内容采用三层结构：明线负责用户追更感，暗线负责技术校准，核心变量负责统一世界观。

| 层级 | 作用 | 执行方式 |
|---|---|---|
| 明线：7 条世界线 | 负责用户可见的系列包装和叙事张力 | 每个章节都归入一个世界线：单机、网络、云、稳定性、信任、算力、AI Native |
| 暗线：IT 基础设施 7 层图 | 负责技术坐标和工程严谨性 | 每篇文章都映射到底层硬件、虚拟化、云平台、运行时、网络、编排或 AI 基础设施 |
| 核心变量 | 负责统一灵魂 | 每篇至少落到控制权、成本、信任、调度、稳定性、平台化或 AI Native |

### 明线：7 条世界线

| 世界线 | 用户可见命题 | 用途 |
|---|---|---|
| 单机时代 | 机器即边界 | 讲 Root、Linux 权限、Kernel、PAM、LSM、最小权限和本地可信 |
| 网络时代 | 连接让世界失控 | 讲 LAN、域控、Kerberos、VPN、Web 安全、API 化和边界失效 |
| 云上帝国 | 控制权转向平台 | 讲虚拟化、资源池、Kubernetes、公有云、私有云、混合云、云中立 |
| 稳定性战争 | 用成本购买确定性 | 讲容量、峰值、限流、降级、SLA、可观测性、FinOps |
| 重建信任 | 身份成为新边界 | 讲 IAM、OAuth、JWT、Zero Trust、Workload Identity、Agent 身份、模型保护 |
| 算力之争 | AI 重构基础设施 | 讲 GPU、网络、电力、数据中心、推理成本、AI Fabric、资源调度 |
| AI Native | 软件开始变成智能体 | 讲 Agent Runtime、MCP、AI Gateway、Tool Protocol、记忆、权限、自治软件 |

### 暗线：IT 基础设施 7 层图

| 层级 | 技术层 | 校准作用 |
|---|---|---|
| 1 | Linux / 裸金属 / 硬件层 | 校准单机、硬件、Kernel、GPU、NUMA、服务器物理限制 |
| 2 | 计算与存储虚拟化层 | 校准 VM、Hypervisor、存储虚拟化、资源池化、隔离 |
| 3 | 云平台 / 基础设施管理层 | 校准 IaaS、PaaS、云控制面、私有云、公有云、混合云 |
| 4 | 容器运行时层 | 校准 containerd、CRI-O、runc、gVisor、Kata、OCI |
| 5 | 网络与服务网关层 | 校准 CNI、Service Mesh、API Gateway、Zero Trust、AI Gateway |
| 6 | 应用编排层 | 校准 Kubernetes、GitOps、IDP、Service Catalog、Platform Engineering |
| 7 | AI 基础设施与模型服务层 | 校准 GPU 调度、AI Fabric、vLLM、Ray、Model Serving、Agent Runtime |

## 全书主线

《现代 IT 基础设施演进史》的核心判断是：

> 技术演进不是工具越来越多，而是业务增长之后，系统复杂度不断超过人的直接控制能力，于是人类不断把控制权交给更高层的平台。

全书围绕四条线索展开：

| 线索 | 关注的问题 |
|---|---|
| 技术演进 | 从单机、网络、虚拟化、云平台、平台工程到 AI 基础设施 |
| 商业压力 | 增长、成本、效率、稳定性、交付速度如何反向塑造技术架构 |
| 组织焦虑 | 权限、协作、故障、合规、跨部门边界如何变成基础设施问题 |
| 控制权迁移 | 控制权如何从人转向脚本、平台、控制面，最终走向 Agent |

## 写作规则

每篇文章必须回答一个现实问题，而不是解释一个工具名。

| 模块 | 要求 |
|---|---|
| 现实钩子 | 从业务增长、组织焦虑、成本压力或技术冲突开场 |
| 技术史脉络 | 说明这个问题为什么在当时出现 |
| 系统边界 | 解释技术解决了什么，又把什么复杂度转移给下一层 |
| 商业含义 | 回到成本、效率、稳定性、控制权或企业能力 |
| 一句话收束 | 回到“复杂系统如何变成企业能力”的主线 |
