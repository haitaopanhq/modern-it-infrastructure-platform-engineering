# 03-04: 一图看懂云中立架构

## 核心观点

云中立不是拒绝使用公有云，也不是把所有能力都压到最低公共分母，而是让组织在关键层面保留选择权。真正的云中立架构，目标不是“任何系统随时搬到任何云”，而是明确哪些能力必须可迁移、哪些能力可以使用厂商托管、哪些能力需要通过平台控制面屏蔽差异。

如果把云中立理解成“不用任何云厂商特性”，系统会牺牲效率；如果把云中立理解成“同时买几家云”，组织会得到更多复杂性。更现实的判断标准是：当价格、合规、容量、地域或战略发生变化时，业务能否以可控成本迁移、扩展或替换关键能力。

## 演进脉络

第一阶段的云中立来自私有云时代。企业希望用 OpenStack、VMware 或自建平台减少对单一厂商的依赖，但那时主要管理的是虚拟机，抽象边界粗，自动化能力有限。

第二阶段由容器推动。OCI 镜像和 Kubernetes API 让工作负载有了更统一的打包、调度和服务发现模型，云中立从“搬虚拟机”变成“搬应用声明”。

第三阶段是平台工程。企业发现 Kubernetes 只解决了部分运行时问题，身份、网络、数据、成本、合规和开发体验仍然散落在不同云控制面。于是云中立开始从技术口号变成组织能力：用内部开发者平台、服务目录和策略治理来管理差异。

AI 基础设施又带来新变量。GPU 供应、模型服务、数据安全和训练成本迫使企业跨云、跨区域寻找容量，但 AI 工作负载对网络、存储和调度的依赖更强，云中立必须从一开始就设计数据和算力边界。

## 关键技术栈

| 层次 | 关键技术 | 作用 |
|------|----------|------|
| 工作负载标准 | OCI Image, Kubernetes, Helm, Kustomize | 统一打包、部署和变更 |
| 基础设施声明 | Terraform, Pulumi, Crossplane | 记录资源意图并降低手工依赖 |
| 平台交付 | Argo CD, Flux, Backstage, Service Catalog | 把多环境能力交给平台治理 |
| 网络与存储 | CNI, CSI, Gateway API, S3-compatible API | 定义迁移时最难替换的接口 |
| 身份与策略 | OIDC, SAML, RBAC, OPA, Kyverno | 保持权限模型和合规规则一致 |
| 可观测性 | OpenTelemetry, Prometheus, Grafana, Loki | 让运行事实脱离单一云控制台 |

这些技术的价值不在于“开源”二字，而在于它们能形成可验证的边界。只要边界没有被测试过，可移植性就只是文档承诺。

## 误区与现实

误区一是把云中立等同于完全不使用托管服务。现实中，适度使用托管数据库、对象存储或消息服务可以节省大量运维成本，关键是提前设计数据导出、备份格式、连接层和替换路径。

误区二是认为 Kubernetes 自动带来云中立。现实是 Ingress、负载均衡、存储卷、节点池、日志、密钥、镜像仓库和账号权限都会把业务重新拉回具体云环境。

误区三是认为云中立越彻底越好。过度抽象会让平台团队维护一套“自研云”，既吃不到厂商能力，又承担全部复杂度。云中立应服务于业务弹性，而不是成为新的工程教条。

## 最佳实践

先定义迁移等级。核心交易、身份、数据、模型资产等能力应具备高等级可迁移性；临时分析、低风险内部工具可以接受更强厂商绑定。

再定义平台接口。业务团队提交的是应用、数据库等级、访问策略、告警规则和容量需求，而不是直接操作每家云的原生资源。平台团队负责把这些意图映射到不同环境。

同时建立定期演练。备份恢复、跨云部署、只读降级、DNS 切换、镜像重建、监控重接入都需要跑过。没有演练的云中立，只是架构图上的中立。

最后保留清晰例外。某些场景为了性能、成本或合规必须使用厂商能力，应在 ADR 中记录原因、退出条件和替代方案，避免沉默锁定。
