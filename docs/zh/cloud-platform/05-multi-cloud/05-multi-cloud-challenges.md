# 03-05: 一图看懂多云真正的问题

## 核心观点

多云真正的问题不是“会不会同时使用 AWS、Azure、GCP 或私有云”，而是组织能否管理多个控制面带来的身份、网络、数据、成本、合规和责任边界。很多团队把多云当成技术选型，最终却发现最难的不是把应用部署到第二朵云，而是让第二朵云进入同一套治理、审计、监控、预算和故障响应体系。

所以多云不是天然的高可用，也不是自动的议价能力。没有统一控制面，多云会把复杂性成倍放大；没有数据策略，多云会被数据重力拖住；没有组织分工，多云会让每个团队都在重复学习不同云的细节。

## 图表结构

```text
多云目标
  - 降低单点供应商风险
  - 获取不同区域和容量
  - 优化成本与采购议价
  - 满足合规和客户要求
        |
        v
真实挑战
  - 身份: 多账号、多租户、多权限模型
  - 网络: 跨云互联、DNS、延迟、出口费用
  - 数据: 同步、备份、一致性、主权约束
  - 运营: 监控、审计、SRE、值班、变更
  - 成本: 标签、分摊、预留、浪费治理
        |
        v
可控多云
  - 平台抽象 + 策略治理 + 演练机制
```

这张图要表达的是：多云的入口是业务目标，难点却集中在治理层。只比较云厂商功能清单，会低估身份、网络和数据的系统性成本。

## 演进脉络

第一阶段的多云主要来自采购和容灾。企业希望避免被单一厂商锁定，于是把备份、灾备或部分非核心业务放到另一朵云上。

第二阶段是云原生普及之后的主动多云。Kubernetes、Terraform、容器镜像和自动化流水线让跨云部署看起来可行，于是多云被包装成“天然可移植”的架构目标。

第三阶段是现实回归。团队发现跨云网络专线昂贵，云数据库迁移困难，日志和权限模型不统一，成本账单难以归因，真正的复杂度不在部署，而在长期运营。

第四阶段是 AI 和全球化带来的必要多云。GPU 容量、数据主权、区域合规和客户落地要求让企业不得不使用多个环境。此时多云不再是口号，而是必须被平台化、制度化管理的运行事实。

## 关键技术栈

| 问题域 | 关键工具与能力 | 关注点 |
|--------|----------------|--------|
| 基础设施声明 | Terraform, Pulumi, Crossplane | 不同 provider 的语义差异和状态管理 |
| 部署交付 | Argo CD, Flux, Helm, Kustomize | 多集群、多区域发布一致性 |
| 身份权限 | OIDC, SSO, IAM Federation, RBAC | 人、服务账号和自动化凭证边界 |
| 网络连接 | VPN, Direct Connect, ExpressRoute, Cloud Interconnect, DNS | 延迟、带宽、出口费和故障域 |
| 数据治理 | Object Storage, CDC, Backup, Velero, Rclone | 数据位置、恢复目标和格式可读性 |
| 成本运营 | FinOps, Kubecost, 标签策略, 预算告警 | 跨云成本分摊和浪费发现 |

多云技术栈的关键不是“工具更多”，而是所有工具都必须进入同一套平台流程。否则每新增一朵云，就新增一个孤岛。

## 误区与现实

误区一是认为多云能天然避免宕机。现实是跨云高可用需要数据复制、DNS 切换、流量调度、回滚演练和一致的故障指挥。没有这些，多云只是多了一个备用环境。

误区二是认为多云一定更便宜。现实是重复建设网络、监控、权限和运维能力会产生隐藏成本；跨云流量出口费和人力学习成本经常抵消采购议价收益。

误区三是认为统一工具可以抹平所有差异。Terraform 可以统一声明方式，却不能消除每家云的网络语义、IAM 模型、托管服务行为和配额限制。

## 最佳实践

从业务理由开始，而不是从技术口号开始。每一朵云都应有明确职责：容量补充、区域合规、灾备、客户托管、AI 训练或成本优化。

建立跨云控制面。账号、标签、预算、日志、审计、镜像、密钥、网络域、告警和事件响应要有统一基线，不能让各云团队各自定义。

控制数据移动。先决定主数据在哪里、哪些数据可以复制、复制频率是什么、恢复目标是什么，再决定应用是否跨云部署。

持续演练退出路径。多云价值来自可选择性，而可选择性必须用演练验证：从镜像重建到数据库恢复，从 DNS 切换到权限回收，都要有可执行流程。
