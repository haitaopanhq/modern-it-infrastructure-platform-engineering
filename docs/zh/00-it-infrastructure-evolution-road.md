# IT 基础设施演进之路：从硬件、云原生到 AI Native

这不是一本按工具分类的教程。

如果把过去二十多年的基础设施演进真正串起来看，整个 IT 世界一直在围绕同一个问题反复变化：当业务越来越快、系统越来越大、团队越来越多、成本越来越高时，企业如何继续扩张，而不被自己的技术复杂度拖住。

很多技术名词后来被拆开讲：虚拟化、云计算、容器、Kubernetes、Service Mesh、GitOps、平台工程、可观测性、AI Infra、Agent Runtime。可它们并不是彼此割裂的技术清单，而是一条连续的技术史。每一代基础设施，表面上是在引入新工具，本质上是在重新分配抽象层、控制权、数据面和治理责任。

换句话说，过去二十多年的 IT 基础设施演进，本质上一直在解决同一个问题：如何让业务持续突破资源边界继续扩张。从虚拟化、云计算，到 Kubernetes、云原生，本质上都是“资源抽象”和“统一调度能力”的不断升级。Kubernetes 的成功，并不只是容器编排，而是它第一次把基础设施真正变成了“分布式操作系统”。但 AI 时代开始之后，传统 Cloud Native 的边界正在被重新暴露。因为 AI Infra 关心的，已经不只是服务扩缩容，而是 GPU 调度、显存拓扑、RDMA、KV Cache 和推理吞吐。

但资源效率只是这条历史的一半。另一半是信任。最早企业相信机器，后来相信内网，再后来相信账号、Token、身份系统和策略控制面。虚拟化、云平台和 Kubernetes 让资源可以被抽象和调度；IAM、零信任、工作负载身份和审计体系，则让每一次访问、每一次部署、每一次工具调用都能够被证明、被约束、被撤销。到了 AI Agent 时代，基础设施必须回答的不只是“任务跑在哪里”，还包括“谁正在代表你行动”。

## 一条更长的演进路径

现代基础设施越来越像这样一条路径：

```text
Hardware
  -> Virtualization
  -> Cloud Control Plane
  -> Runtime
  -> Network
  -> Orchestration
  -> Platform Engineering
  -> Trust / Identity Control Plane
  -> AI Infra
  -> Agent Control Plane
```

## 资源抽象这条暗线

很多人以为，现代基础设施的核心是 Kubernetes、Docker，甚至 AI。但如果把过去二十多年 IT 演进真正串起来看，会发现行业持续演进的主线，一直是“资源抽象”：把离散硬件逐渐抽象成统一资源池，再把复杂系统逐渐抽象成可编排能力。

最早解决资源共享问题的，并不是云，而是 Kernel。大型机和 UNIX 时代已经开始回答一个基础问题：如何让一台机器同时服务多个任务。后来 PC 时代出现 DOS、macOS、Windows、Linux，本质上也都在继续演进这套资源管理逻辑。进程调度、虚拟内存、page cache、fork、swap，都是 Kernel 在把 CPU、内存、IO 从物理硬件抽象成可共享资源。很多人后来讨论云计算，却忽略了 Kernel 本身才是现代资源抽象体系真正的起点。

互联网规模扩大之后，行业进入 Hypervisor 时代。KVM、VMware、Xen、OpenVZ 让服务器第一次从一台具体机器，变成可以切分、隔离、迁移和回收的资源池。虚拟化真正改变的，不只是性能曲线，而是资源管理方式：CPU、内存、网络、存储开始脱离单台物理机，进入可调度的系统边界。

再后来，Docker、Kubernetes、CNI、CSI、Ceph 把抽象继续向上推进。Docker 抽象运行环境，CNI 抽象网络，CSI 抽象存储，Ceph 抽象分布式存储池，Kubernetes 抽象整个集群控制面。现代基础设施越来越像一个巨大的资源翻译系统：底层是硬件，上层是 API，中间全部是抽象层。

应用架构也在同一条压力链上变化。最早的单体应用并不落后，它只是更适合业务起步阶段：一个工程、一套数据库、一台服务器，开发快、部署简单、成本低。真正推动拆分的不是技术潮流，而是增长。用户变多、团队变多、端侧变多、发布频率变高之后，页面、业务、数据库和后台管理全部挤在一个工程里，耦合、发布风险、扩容困难和排障依赖就会集中爆发。

前后端分离让系统先从“一个团队写一个工程”进入“多端、多团队围绕 API 协作”的阶段。服务拆分又把订单、支付、库存、搜索这些业务能力逐渐拆成可以独立扩容和独立发布的服务。注册中心出现后，系统从写死地址进入动态治理：服务自动注册、自动发现、自动摘除、自动扩缩容。到了分布式架构阶段，系统获得了更强的扩展性和容错能力，也付出了链路变长、事务复杂、网络不稳定和排障困难的代价。

这也是 Service Mesh 与云原生架构出现的背景。行业曾经希望把流量治理、熔断限流、安全认证、灰度发布统一下沉到基础设施层，但现实证明，治理能力下沉并不会让复杂度消失，只会把复杂度集中到新的控制面。Kubernetes、CI/CD、网关、可观测性、配置中心和服务治理组合在一起之后，现代架构真正进入平台化治理阶段。架构不再只是开发团队的问题，而是业务、研发、运维、安全和平台共同参与的工程体系。

所以，从单体到分布式，再到云原生，本质上不是系统越来越“高级”，而是在“拆分复杂度”和“重新治理复杂度”之间循环。AI 时代之后，这个问题会再次被放大：下一代架构究竟会继续分布式化，还是重新回到某种超级统一平台，还是让 Code Agent 直接理解、重构和迁移几十万行老代码，都取决于企业能否把自治能力放进可验证、可审计、可回滚的控制面里。

但抽象从来不是为了消灭硬件，而是为了更高效地使用硬件。AI 时代之后，行业又重新开始看见 GPU、NUMA、RDMA、NVLink、PCIe、DPU。它们重新决定系统上限，是因为资源抽象走到高密度算力阶段之后，任何隐藏得过深的硬件差异，都会重新以吞吐、延迟、显存、拓扑和成本的形式回到系统设计里。

所以这本书讨论的不是一组工具的替换史，而是一场持续二十多年的资源抽象运动。Kernel 把硬件变成可共享资源，Hypervisor 把服务器变成资源池，云平台把资源供给变成 API，Kubernetes 把集群变成声明式控制面，AI Infra 又把抽象重新拉回硬件效率和高速互联。现代基础设施的本质，不是技术堆叠，而是资源抽象、控制权迁移和治理能力持续重组。

最早的时代，基础设施的核心从来不是软件，而是硬件。CPU、内存、磁盘、网络带宽直接决定系统上限。互联网业务增长之后，企业第一次发现，限制业务扩张的不只是代码，而是资源无法被灵活利用。于是虚拟化出现了。它真正改变世界的地方，不只是多跑几台虚拟机，而是让计算资源脱离具体物理机器，变成可以分配、迁移、回收、调度的资源池。

这也是为什么“容器淘汰虚拟化”这个说法经不起历史检验。Intel VT-x、AMD-V、EPT/NPT、IOMMU、SR-IOV、PCIe Passthrough 和 vGPU 这些能力持续成熟之后，虚拟化并没有退场，而是越来越贴近 CPU、内存和设备本身。Docker 与 Kubernetes 把隔离和调度推到进程与应用层，AI Infra 又把系统重新拉回 PCIe 拓扑、NUMA、RDMA、NVLink、显存带宽和 GPU Direct。表面看路线在变化，底层问题一直没变：如何在保留资源抽象、隔离和调度能力的同时，尽可能降低虚拟化损耗。

后来这种抽象继续扩张。Ceph 把存储池化，Open vSwitch 把网络虚拟化，SR-IOV 与 DPDK 重新定义高性能网络路径，CSI、CNI、CRI 逐渐形成标准接口。硬件不再只属于某一台机器，而开始属于整个数据中心。

存储虚拟化是这条暗线里最容易被低估的一段。很多人以为存储只是“硬盘”，但它的演进同样是一场资源抽象史：本地磁盘把数据绑定在单台服务器上，SAN/NAS 把存储从服务器里抽离出来，LVM、Thin Provision 和 Snapshot 把磁盘变成逻辑卷，Ceph、GlusterFS、HDFS 把离散磁盘抽象成统一存储池，云硬盘、对象存储、文件存储和归档存储又把存储变成服务。云原生时代之后，CSI、Rook、Longhorn、OpenEBS 继续向上抽象，让存储开始与 Kubernetes 生命周期绑定。AI 时代则重新改变目标：传统数据库时代拼 IOPS，AI Storage 拼吞吐、数据流和 GPU Feeding。NVMe-oF、GPUDirect Storage、Checkpoint，本质上都在解决同一个问题：GPU 不能等待 IO。

网络虚拟化则是另一条经常被误解的主线。很多人以为，网络只是“连接机器”。但如果真正回看过去二十年的基础设施演进，会发现网络其实一直在从“连接系统”逐渐变成“数据面系统”。最早的传统网络时代，核心是 VLAN、Router、Switch、STP。那个时代网络以硬件设备为中心，主要解决南北向流量与用户访问问题，网络几乎是静态规划，扩展能力有限。

后来 Overlay Network 开始出现。VXLAN、GRE、Open vSwitch 让网络第一次真正拥有“虚拟化能力”。网络不再完全依赖物理拓扑，而开始通过隧道与虚拟交换机实现逻辑隔离。网络第一次开始脱离物理交换机，形成可以跨主机、跨机架、跨资源池移动的逻辑网络。

再后来，行业进入 SDN 与 NFV 时代。OpenFlow、NSX、SDN Controller 开始把控制面与数据面分离；防火墙、负载均衡、网关、EPC 等网络设备，也开始从专用硬件转向软件化网络功能。网络第一次真正进入“可编排时代”：策略可以下发，流表可以修改，网络功能可以部署、扩缩和迁移。

随着云计算与高性能网络继续发展，行业又开始重新关注数据面性能。DPDK 出现后，网络大规模进入“用户态高速转发”时代。过去大量网络包需要经过 Linux Kernel 网络栈，而 DPDK 通过用户态 Polling、HugePage、零拷贝与绕过内核协议栈，显著降低转发延迟。于是虚拟交换机、NFV、vRouter、高性能网关开始真正具备接近专用设备的吞吐能力。

Kubernetes 普及后，CNI、Flannel、Calico、Cilium、eBPF 又继续把网络推进到云原生时代。网络开始不只是转发流量，而是承担服务发现、安全、可观测性与策略治理。eBPF 让 Linux Kernel 的网络数据面第一次真正变得“可编程”；XDP 更进一步，把部分过去在用户态或专用设备里的能力，重新带回 Kernel Datapath，并逐渐靠近 NIC 与硬件卸载层。

AI Infra 出现后，网络再次发生巨大变化。传统互联网以南北向流量与 Scale-Out 为主，而 AI Cluster 开始变成典型的东西向通信与 Scale-Up 系统。GPU 与 GPU 之间需要大量同步通信，于是行业重新追求低延迟、高吞吐、零拷贝的高速互联能力。HPC 世界长期使用 InfiniBand，后来 RDMA 成为核心通信模型，RoCE 把 RDMA 带入以太网体系，NVLink 负责 GPU 之间的本地高速互联，NCCL 则在这些高速通信能力之上，实现 GPU 集群间的并行通信与 AllReduce。

这条线说明，AI 时代真正开始让网络从“请求转发”变成“算力互联”。它不再只是应用访问路径的一部分，而是 GPU 利用率、训练效率、推理吞吐和系统成本的组成部分。网络虚拟化的终点不是把网线藏起来，而是把连接、策略、数据面和算力互联重新组织成平台能力。

资源池化之后，一个更大的问题出现了：谁来管理这些资源。于是 Cloud Control Plane 到来。AWS、OpenStack、Harvester 这些平台真正重要的地方，不只是底层虚拟化，而是把基础设施变成统一控制平面。创建机器、分配网络、挂载存储、调度资源，从此进入 API 驱动时代。

Kubernetes 出现后，行业又进入一个新的中间层时代。它像一种分布式容器操作系统，统一了 Runtime 与 Orchestration 之间的关系。Mesos、Docker Swarm 逐渐退场，Kubernetes 成为事实标准。但任何成功到极致的中间层，最后都会暴露自己的边界。数据库跑 K8s，AI 跑 K8s，边缘跑 K8s，虚拟机跑 K8s，存储也跑 K8s，很多企业最后发现，自己管理的已经不是业务，而是 Kubernetes 本身。

AI Infra 正在重新打破这个结构。传统 Web Infra 关注 HTTP 请求、副本数、弹性扩缩容和服务治理；AI Infra 更关注 GPU 调度、KV Cache、显存拓扑、RDMA、Tensor Parallel、Pipeline Parallel、Token 延迟、多模型路由和推理吞吐。模型规模进入 30B、70B、100B+ 后，真正决定系统上限的，重新变成硬件利用率和高速通信能力。它们已经不是同一种问题。

所以 vLLM、Ray、TensorRT-LLM、SGLang、Triton 开始快速崛起。AI 时代并没有让硬件变得不重要，反而让 GPU、NVLink、HBM、RDMA、高速网络、Linux Kernel、eBPF、cgroups 重新成为系统上限。基础设施又回到那个最原始的问题：如何让硬件资源被更高效地利用。

未来 Kubernetes 不会消失，但会逐渐“下沉”，像 Linux 一样成为基础设施底座。真正往上生长的，将是 AI Gateway、GPU Scheduler、Inference Fabric、Semantic Cache、Agent Runtime。Cloud Native 没有结束，它正在进入下一阶段：AI Native。

## 本书的视角

这本书不把基础设施写成安装手册，而把它放回企业真实语境中理解。

| 视角 | 这本书关心的问题 |
|------|------------------|
| 技术 | 每一层抽象解决了什么系统边界问题 |
| 商业 | 业务增长、成本结构和交付速度如何推动架构变化 |
| 焦虑 | 团队为什么会害怕不可控、不可见、不可审计和不可回滚 |
| 复杂度 | 新工具为什么既降低复杂度，又制造新的复杂度中心 |
| 控制权 | 控制权如何从人转向脚本、控制面、Git、平台和 Agent |

已经发布的图文选题，正好构成了这本书的现实入口：`一图纵览IT基础设施演进之路` 讨论总体路径，`业务部门和基础 IT 部门，到底在关注什么？` 讨论组织语言差异，`一图看懂从单体应用到分布式架构` 讨论业务增长如何逼出系统拆分与治理复杂度，`一图读懂现代网络与协议` 讨论连接如何变成数据面，`一图读懂监控的前生今世` 讨论系统不可知带来的治理焦虑，`一图纵览账户与安全体系` 讨论安全边界如何从网络迁移到身份，`AI 会重新定义下一代基础设施吗？` 则把这条线推向 AI Native 与 Agent 控制面。

## 阅读方式

前七章是这本电子书的主干。

第 1 章先建立全局观：现代 IT 最大的问题已经不是功能，而是复杂度。第 2 章进入网络：连接不再只是把服务连起来，而是逐渐变成系统控制与数据面。第 3 章讨论数据库：它不再只是 MySQL、建表和 SQL，而是从交易一致性、实时分析、缓存、数据流动、向量检索一路裂变成 AI 基础设施里的核心层。第 4 章讨论存储：数据留存、访问模式、成本和 AI 训练让存储重新成为基础设施核心。第 5 章讨论可观测性：当系统变得不可直觉理解，监控就必须升级为治理能力。第 6 章讨论 DevOps 到平台工程：控制权如何从人工、脚本、流水线、GitOps 迁移到平台。第 7 章讨论平台工程核心能力：复杂度如何被产品化，最终变成组织可以复用的企业能力。

第 10 章把安全接回同一条主线：从本地权限、企业目录、Web Token、云 IAM、零信任，到 AI Agent 身份和模型能力保护，账户与身份安全演进史就是信任边界不断迁移的历史。它说明现代基础设施不只是把资源变成可调度对象，也必须把信任变成可验证、可撤销、可审计的控制面。

后续章节会继续把运行时、故障治理、架构决策、实验环境、平台闭环和 Agent 工程接上这条主线。它们共同回答同一个问题：

> 基础设施演进不是工具越来越多，而是抽象、控制权、数据面和治理机制不断重组，最终把复杂系统转化为企业能力。
